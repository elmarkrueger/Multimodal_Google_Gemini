# **Product Requirements Document: Architecture and Implementation of a Multimodal Google Gemini Custom Node for ComfyUI**

## **1\. Introduction and strategic classification of hybrid AI workflows**

The rapid technological evolution of generative artificial intelligence has led to a significant diversification of user interfaces and orchestration tools in recent years. At the heart of this development, ComfyUI has established itself as the dominant standard for the node-based construction of complex inference pipelines.1Originally, this platform was primarily designed for running local latent diffusion models such as Stable Diffusion, SDXL or Flux, which is why the core architecture is heavily focused on manipulating tensors in local graphics memory (VRAM).1Nevertheless, the software is increasingly evolving into a hybrid architecture, in which local computing power is augmented by high-performance, cloud-based API services.1This development enables users to directly integrate the superior semantic capabilities, deep logical reasoning, and massive context windows of Large Language Models (LLMs) and Vision-Language Models (VLMs) into their image- or video-generating workflows.1

This Product Requirements Document specifies the architecture, technical logic, and mandatory code elements for developing a highly integrated, natively multimodal Google Gemini custom node for ComfyUI. The primary objective of this specific node is to analytically analyze image, audio, and video data and generate highly complex image prompts for downstream diffusion models.2To bridge the gap between local tensor processing and cloud-based inference, the architecture utilizes the modern google.genai Python Software Development Kit, which provides a universal, highly optimized interface for the Gemini Developer API and completely replaces outdated libraries.2

A critical requirement for this system component is the absolute guarantee of flexibility while simultaneously maintaining security. The node requires the input of a user-specific API key, which, for security reasons, must be hidden within the ComfyUI graphical user interface to prevent unintentional leaks when sharing workflows.2Furthermore, strict support for optional inputs is required, especially for a separate video input that is used exclusively as a stylistic reference during prompt creation.5The ability to dynamically set model-specific parameters such as the "Thinking Mode" for deeper logical reasoning and basic inference metrics transforms this node from a simple API wrapper to a central cognitive router within the ComfyUI pipeline.7This report provides the exhaustive technical blueprint that an autonomously operating coding agent needs to implement the node flawlessly, in a memory-efficient manner, and in strict compliance with global API restrictions.

## **2\. Fundamentals of ComfyUI architecture and Node declaration**

Developing a custom node for ComfyUI requires a deep understanding of the underlying Directed Acyclic Graph, which determines the execution order of the visual pipeline. Each node in ComfyUI is defined as a self-contained Python class that must provide specific class attributes and methods so that the system can recognize, render, and execute it.5The essential attributes that the coding agent must implement in the structure of the class include INPUT\_TYPES, RETURN\_TYPES, RETURN\_NAMES, FUNCTION and CATEGORY.5

The INPUT\_TYPES attribute acts as the primary interface between the Python backend and the JavaScript frontend. It is a mandatory class method that returns a dictionary specifying the expected data types and widgets.5Within this dictionary, a strict distinction is made between required and optional.5Parameters defined in the required block create either a fixed widget (such as a text field or a slider) or an input pin on the user interface, which must be supplied with data in order for the graph to start execution.5The optional category, however, is of paramount importance for the multimodal flexibility of this specific Gemini node. Inputs for images, videos, or audio files declared here generate input pins on the node, but allow the workflow to execute even if they remain unconnected.5In the runtime environment, this means that the function in the backend only receives these parameters via the \*\*kwargs system or as named arguments if a data stream is actually present.5

The RETURN\_TYPES attribute defines the data types that the node passes to downstream nodes after successful execution.5Since the primary task of the designed Gemini Node is semantic analysis and the generation of instructions for diffusion models, this attribute must be declared as a tuple containing the type STRING.4The RETURN\_NAMES correspond to these types and determine the labels of the output pins on the graphical user interface. The FUNCTION variable contains the name of the core method within the class (for example, generate\_gemini\_response) that is called as soon as the ComfyUI scheduler reaches the node.5Finally, the CATEGORY attribute ensures the correct hierarchical sorting of the node into the context menu of the user interface, so that users can find the module intuitively.

## **3\. Definition of node interfaces and structural parameters**

To fully implement the functionalities specified in the requirements profile, the coding agent must construct the INPUT\_TYPES method with the highest precision. The dictionary must contain all text fields, dropdown menus for API settings, and dynamic media pins. The input architecture reflects the balance between necessary configuration data and the desired multimodal variability.1

The following table specifies the architecture of the mandatory inputs to be implemented in the required block of the interface declaration:

| model | (list, ) | A dropdown menu for selecting the model architecture. The list should include models available in the Free Tier, such as gemini-3.1-flash-lite-preview, gemini-3-flash-preview, and gemini-2.5-flash.1 |
| system\_prompt | ("STRING", {"multiline": True}) | A multi-line text field that dictates the fundamental role, system boundaries, and expected format of the model response. This prompt is passed to the API's System Instruction interface.9 |

This architectural separation ensures that the node's user interface remains clear and users only need to configure the parameters relevant to their specific use case, be it pure text processing or complex video analysis.

## **4\. Implementation of the graphical user interface and API key masking**

An essential and often underestimated security requirement in the development of ComfyUI custom nodes is the protection of sensitive access data. The standard ComfyUI architecture converts all parameters of type STRING into regular HTML text fields, the content of which is freely visible on the screen.4When users share screenshots of their workflows or transmit their screen in collaborative environments, there is a significant risk of their Google Gemini API key being compromised. The google.genai SDK requires this key for authentication, making a robust masking solution essential.2

Since the Python backend does not offer native data types for password-protected strings, the coding agent must solve this problem by interfering with ComfyUI's JavaScript frontend.4The system allows developers to dynamically manipulate the Document Object Model (DOM) as soon as a node is created in the web interface. For this to work, a separate JavaScript file (for example, hide\_api\_key.js) must be created in a dedicated web directory for the custom node.4In the Python initialization file (\_\_init\_\_.py), the system is instructed by the declaration WEB\_DIRECTORY \= "./js" to deliver this script to the client's browser when the server starts.4

The internal logic of the JavaScript module uses the \`app.registerExtension\` method. Within this extension, the \`nodeCreated(node)\` hook is used, which is called asynchronously every time a new node is placed on the canvas or loaded from a saved workflow.4The code must first verify whether the \`node.comfyClass\` attribute matches the internal class name of the Gemini node being developed. If this check is successful, the script iterates over the \`node.widgets\` array to identify the specific control named \`api\_key\`.4

Once this widget is located, the script accesses the underlying DOM structure. Every string widget in ComfyUI has a reference to the native HTML input element, which is typically accessible via the \`inputEl\` attribute. By simply reassigning \`widget.inputEl.type \= "password"\`, the browser is forced to immediately convert the string's representation into escape characters (periods or asterisks).4The actual value of the string remains unchanged in the application's memory and is correctly transmitted to the Python backend when the graph is executed. This architectural measure ensures that the API key is never visible in plaintext, thus fully complying with the specification's security requirements.

## **5\. Integration of the google.genai Python SDK and authentication**

The technological core of the custom node is manifested in the correct integration of the google.genai Software Development Kit. Historical analyses show that developers in the past often used outdated libraries like google-generativeai, which were fragmented and exhibited incompatibilities between AI Studio and the enterprise environment Vertex AI.2The architecture of this node must necessarily use the modern, unified google.genai package, which serves as a universal abstraction layer.3The coding agent is therefore instructed to place the corresponding import instructions (from google import genai and from google.genai import types) at the top of the backend logic.3

The client is initialized dynamically with each iteration of the FUNCTION method, as the API key entered by the user could change between different calls. The instantiation process is extremely minimalist thanks to the new SDK.2The code extracts the value of the api\_key widget and passes it directly to the constructor: client \= genai.Client(api\_key=api\_key).15It is critical that the agent implements basic error checking at this point. If the passed string is empty, execution must be immediately interrupted by throwing an informative exception. This protects the system from network timeouts and provides the user with clear feedback via the ComfyUI interface that the authentication information is missing.

The instantiated client object now serves as the central hub for all network communication. It abstracts the complex HTTP REST specifications, handles the correct serialization of the JSON payloads, and manages polling during asynchronous calls.2By using this central object, the node is able to send simple text prompts as well as orchestrate heavyweight media uploads via the integrated Files API, forming the basis for the required native multimodality.2

## **6\. Configuration of the model parameters and the cognitive "thinking" mode**

A key feature of modern LLM integrations is the fine-grained control over the model's generation behavior. The specification explicitly requires that model parameters, and in particular the advanced "thinking mode" of the Gemini architecture, be controllable via the node. The google.genai SDK bundles these settings into a strictly typed Pydantic class called GenerateContentConfig, which is passed as a parameter to the inference method.9

The basic parameters control the stochastic nature of the text generation. The \`temperature\` parameter regulates the degree of randomness in token selection. A value of 0.0 forces the model to produce maximally deterministic responses, which is advantageous for structured data extraction, while values ​​around 1.0 allow for more creative and diverse prompt extensions.8The top\_p value (Nucleus Sampling) changes the selection by considering tokens only until the sum of their probabilities reaches the configured value.8Similarly, top\_k limits the selection to a fixed number of the most probable tokens.8The coding agent must create an instance of types.GenerateContentConfig and populate these three variables with the values ​​configured by the user via the widgets.

Additionally, separating context and instructions requires the use of system instructions. The content of the \`system\_prompt\` text field cannot simply be concatenated with the user prompt. Instead, the SDK provides the option to pass this as a dedicated \`system\_instruction\` object within the \`GenerateContentConfig\`.9This strict separation minimizes the risk of prompt injection attacks and focuses the model's attention on its superior persona, for example, as a professional prompt engineer for diffusion models.

The most architecturally complex aspect of the configuration is the control of the "Thinking Mode" (Deep Reasoning). Models in the Gemini 3 and 2.5 series utilize an internal thinking process that significantly enhances their capabilities in multi-stage planning and logical deduction.18The specification requires a toggle to enable or disable this mode. The SDK implements this via the \`thinking\_config\` attribute, which expects an instance of \`types.ThinkingConfig\`.7In the latest generation of the API, this is primarily controlled by the parameter thinking\_level, which can take values ​​such as low, medium or high.7

The coding agent must implement precise conditional logic for this. Since the user can set the mode to "disabled" in the dropdown menu, the code must handle this state. Ideally, if "disabled" is selected, the \`thinking\_config\` should be omitted entirely or set to a minimum to prevent the API from throwing a 400 error code due to incompatible parameters.7If, however, a valid level such as high was selected, the code instantiates thinking\_config=types.ThinkingConfig(thinking\_level=thinking\_mode) and embeds this object in the GenerateContentConfig.7This dynamic configuration enables the ComfyUI Node to scale seamlessly between lightning-fast, resource-efficient calls and in-depth, time-consuming cognitive analyses.

## **7\. Processing visual data streams: Image tensors and stylistic references**

The fundamental strength of the designed node lies in its ability to analytically process the pixel data from the ComfyUI pipeline. The challenge for the coding agent lies in the mathematical and structural incompatibility between the storage formats of the two systems. ComfyUI transports image data in local memory as multidimensional PyTorch tensors.10A typical image entry has the form![][image1], where ![][image2]for the batch size,![][image3] and ![][image4]for height and width, and![][image5]stands for the RGB color channels.10The values ​​of these tensors are usually expressed as 32-bit floating-point numbers in the range of![][image6]normalized.10

The Google Gemini API, on the other hand, expects visual data either as Base64-encoded byte strings or as uploaded files in standardized formats such as JPEG or PNG.11To close this gap, the custom node's backend must implement a dedicated conversion routine. If the main function detects that the image argument is not equal to None, the tensor must first be converted to the scalar.![][image7]multiplied and transformed into the data type of an 8-bit Unsigned Integer (uint8), usually via a Numpy array.10The first image is then isolated from the batch dimension and converted into a native image object using the Python Imaging Library (PIL) by calling PIL.Image.fromarray().20Fortunately, the modern google.genai SDK completely abstracts the subsequent Base64 encoding; the PIL object can be directly inserted into the list of contents that is later sent to the model.22

One specific requirement of the PRD is the optional integration of a second image input, which is intended to function as a "style reference".5In the practice of generative AI, this is a powerful concept for assigning a visual tone to a language model without it misinterpreting this image as the primary object of study. The code must check whether the \`style\_reference\` pin has been connected in the graph. If so, this second tensor undergoes the same mathematical transformation to the PIL object as the main image.

The crucial architectural trick here lies in so-called in-context learning (prompt engineering). The model requires semantic anchoring to correctly classify the second image. The coding agent must construct the list of contents in a structured manner. First, the main image is injected, followed by a clarifying textual directive from the system, such as:*"The following image is to be used exclusively as a stylistic reference for analysis and generation."*Only after this string is the PIL object of the style reference inserted into the array.6This sequential and explicitly labeled arrangement ensures that the native multimodality of the Gemini model flawlessly processes the conceptual separation between the primary object of analysis and the stylistic model.6

## **8\. Processing of temporal and auditory data: Audio and video via the Files API**

While individual images can be efficiently processed in main memory and passed to the REST endpoints as inline data, high-dimensional audio and video data streams inevitably push these architectural boundaries.2A key feature of Google's Free Tier architecture is the limitation of inline payloads, which is why the integrated Files API of the SDK must be used for heavyweight media.2This API allows the asynchronous upload of files up to a size of 2 gigabytes into the Google infrastructure, where they are processed server-side before the actual inference.2

### **Processing of audio tensors**

When the user connects a node like the ComfyUI Load Audio Node to the Gemini Node, the audio argument is populated in the function. In ComfyUI extensions (such as the VideoHelperSuite), audio data is not passed as finished MP3 files, but as complex dictionaries, which usually contain a one-dimensional waveform tensor and an integer for the sample rate (e.g., 44100 Hz).12

Since the Google Files API expects standardized file formats, the backend logic must reconstruct these tensors. The coding agent must create a temporary file (preferably with the .wav extension) on the server's file system using the Python module tempfile.2Using libraries like torchaudio.save or scipy.io.wavfile.write, the tensor is written to this file. Immediately after successful saving, the node initiates the upload process: uploaded\_audio \= client.files.upload(file=temp\_audio\_path).2The SDK returns a file object containing a unique URI, which then serves as the representation of the audio file in the contents array.16

### **Video batch processing**

Video processing presents the greatest technical complexity, as the concept of "video" does not exist natively in ComfyUI's core architecture. Instead, video loaders (such as the VHS\_LoadVideo node) pass videos as gigantic batches of images. A video of just a few seconds in length manifests as a 4D tensor, where the batch dimension![][image2]for example, it takes the value 150 (for 150 frames).12

An attempt to convert 150 high-resolution still images in memory into Base64 strings and simultaneously send them to the API would result in massive performance drops and quota exceedances.2Therefore, the architecture necessarily requires an intermediate recompression step. The coding agent must implement logic that iteratively converts the tensor batch into individual frames and assembles these into a compressed .mp4 video file on the hard drive using libraries such as imageio or by directly calling ffmpeg.25

Similar to audio processing, this temporary video is uploaded via the Files API: uploaded\_video \= client.files.upload(file=temp\_video\_path).2This methodology is fundamental because Google's servers are optimized to perform "Spatio-Temporal Reasoning" (the spatial-temporal analysis of actions) natively and asynchronously based on video files, which minimizes the network load between the ComfyUI server and Google.2

In both cases, strict memory management is an essential component of robust software architecture. The agent must guarantee that the .wav and .mp4 files created by tempfile are completely deleted from the host system in a finally block of Python execution to prevent gradual fragmentation and disk space exhaustion.

## **9\. Assembly of the multimodal prompt and execution of the inference**

After all potential data streams – text, isolated images, uploaded audio URIs, generated video files and explicit style references – have been transformed into compatible formats, the logic culminates in the assembly of the final contents array.26The modularity of this array is the foundation of Gemini's native multimodality.

The array is built sequentially. First, the code adds the user-generated \`user\_prompt\` as a text component. Then, the script checks for the existence of the optional parameters (\`image\`, \`audio\`, \`video\`, \`style\_reference\`) one after the other. For each existing element, the corresponding object (either a direct PIL image or a returned file object from the Upload API) is appended to the array.15The flexibility of the google.genai SDK allows these different modalities to be mixed in a single, flat list, as the SDK internally handles the construction of the specific Part objects for the REST interface.16

Once the payload is prepared and the configuration objects (GenerateContentConfig including system prompt and thinking parameters) have been instantiated, the synchronous inference method is called:

response \= client.models.generate\_content(model=model, contents=contents, config=config) 9

The model now processes the entire multimodal semantic space in a single pass. The model's response is received by the SDK, deserialized, and made available in the \`response.text\` attribute.18To respect the conventions of the ComfyUI architecture, the method must return this string as a tuple: return (response.text,).5The mandatory comma signals the tuple property to the Python interpreter, which then forwards the string to the defined output pin of the node and makes it available for downstream text processors or CLIP encoders.5

## **10\. Error handling, rate limiting and infrastructural resilience**

Developing a software component that relies primarily on the Google AI Studio Free Tier requires proactive consideration of infrastructural system limitations. Google's economic realities dictate that free access is regulated by strict, highly dynamic rate limits (quotas) to control data center utilization.2The quota architecture limits requests via metrics such as RPM (Requests Per Minute) and RPD (Requests Per Day).1

During periods of high global utilization or intensive batch processing by the user in ComfyUI, encountering the HTTP error code 429 (RESOURCE\_EXHAUSTED or Too Many Requests) is a mathematical certainty.1If the node were to crash natively in this scenario, it would terminate the execution of the entire ComfyUI graph, resulting in the total loss of potentially hours of local rendering processes.

To eliminate this architectural weakness, the coding agent is required to implement mechanisms to increase resilience. The established best practice in Python-based data pipelines is the use of the tenacity library.2This library allows the declaration of retry logic using elegant Python decorators.

The core function that makes the API call (client.models.generate\_content) must be enclosed in a helper function that includes a Tenacity decorator. This decorator implements an exponential backoff.2The underlying concept works as follows: If the request fails due to a limit violation or a temporary network timeout, the decorator catches the specific exception (for example, google.genai.errors.APIError). Instead of passing the error to ComfyUI, the thread pauses for a short period (e.g., 2 seconds) and initiates a retry. If this also fails, the wait time is increased exponentially (e.g., to 4, then 8 seconds) until a defined maximum number of attempts is reached.2

This defensive programming strategy almost completely isolates the latency problems and restrictions of the cloud environment from ComfyUI's local, deterministic execution engine. It guarantees that workflows run robustly even under suboptimal network conditions and represents the transition from an experimental script to a production-ready tool.

## **11\. Synthesis of the required libraries and structural code elements**

To guarantee the coding agent a precise implementation template, this final section synthesizes the essential technical dependencies and the basic class specification. The successful execution of this project depends on the consistent integration of these building blocks.

The environment in which ComfyUI runs requires the installation or presence of the following Python libraries:

* google.genai: The modern client for orchestrating all model calls and file transfers.2  
* pydantic: Implicitly used by the SDK for strict typing and validation of configuration objects.2  
* torch: For manipulating the multidimensional image, video, and audio tensors provided by ComfyUI.10  
* numpy: A necessary mathematical bridge layer for converting PyTorch tensors into image storage structures.10  
* Pillow (PIL): For converting arrays into standardized image objects that can be natively integrated into the prompt by the SDK.10  
* tenacity: On the implementation of essential retry logic and exponential backoff.2  
* torchaudio or scipy: For the reconstruction of temporary audio data from raw tensors.13  
* imageio (or an ffmpeg wrapper library): For assembling Tensor batches into standards-compliant .mp4 video files.25

The basic structure of the Python script must define a class, for example, GeminiMultimodalNode. Within this class, the \`@classmethod def INPUT\_TYPES(cls):\` dictates that \`api\_key\`, \`model\`, \`system\_prompt\`, \`user\_prompt\`, \`thinking\_mode\`, \`temperature\`, \`top\_p\`, and \`top\_k\` are placed in the required dictionary. The dynamic media inputs \`image\`, \`style\_reference\`, \`audio\`, and \`video\` are required to be moved to the optional dictionary to ensure flexibility in the graph.5

The method RETURN\_TYPES \= ("STRING",) defines the output, while FUNCTION \= "generate\_response" specifies the name of the execution function.5In parallel, the agent must ensure the creation of the JavaScript module (e.g., in the ./js/ subdirectory). This script registers an extension for ComfyUI, locates the HTML element of the api\_key widget, and hard-codes its type property to "password" to guarantee visual security within the UI context.4

The symbiosis of this structured backend logic, secure frontend manipulation, and robust error handling empowers the coding agent to create a custom node that seamlessly integrates the analytical and cognitive performance of Google Gemini models into ComfyUI's visual ecosystem.
